# BE-06.01 Validation Foundation and DTO Standard

Status: Draft
Phase: BE-06 Validation
Milestone: BE-06.01
Scope: Backend validation foundation and DTO handoff standard
Owner: Backend Validation Steward

## Purpose

Define the validation foundation and DTO standard that BE-06 tasks must follow before normalizing individual modules.

This milestone records repository reality first, then establishes the safe target standard for request validation, DTO construction, and validation error behavior.

## Repository Reality Observed

The current backend repository already contains multiple validation-related foundations:

```text
fieldops-be/src/core/validation/
fieldops-be/src/core/errors/ValidationError.js
fieldops-be/src/shared/validators/
fieldops-be/src/shared/errors/
```

Current observations:

- `src/core/validation/validate.js` validates `body`, `params`, and `query` and writes parsed values back to `req[part]`.
- `src/core/validation/schemaUtils.js` supports schema parsers such as `safeParseAsync`, `safeParse`, `parseAsync`, and `parse`.
- `src/core/errors/ValidationError.js` emits `REQUEST_VALIDATION_ERROR`.
- `src/shared/errors/httpErrors.js` also defines a `ValidationError` with default code `VALIDATION_ERROR`.
- `src/shared/validators/idValue.js` and `textValue.js` throw shared `ValidationError` with codes such as `REQUIRED_ID` and `REQUIRED_STRING`.
- `equipment.validation.js` uses Zod schemas but owns a local `validateBody` helper and local error construction.
- `issue.routes.js` currently has no validation middleware.
- `issue.controller.js` currently forwards raw `req.body`, `req.query`, and route params into services.

## Foundation Decision

BE-06 will treat `fieldops-be/src/core/validation` as the request validation middleware foundation.

Reason:

```text
Route / Controller request validation belongs near HTTP execution.
Core validation already supports body, params, and query.
Core validation already normalizes schema parser behavior.
Core validation already produces a request validation error shape.
```

`fieldops-be/src/shared/validators` remains reserved for reusable primitive validators only when they are domain-neutral and intentionally adopted as shared backend standards.

## Request Validation Standard

Route-level validation should use schema groups by request part:

```js
validate({
  params: paramsSchema,
  query: querySchema,
  body: bodySchema,
})
```

The validator must parse and sanitize request values before the controller calls a service.

Controllers may read parsed values from:

```text
req.params
req.query
req.body
```

Controllers must not perform ad-hoc request-shape validation except for minimal orchestration or unavoidable compatibility handoff.

## DTO Handoff Standard

Services should receive explicit DTO-style inputs, not raw Express request objects.

Preferred controller-to-service shape:

```js
service.performAction({
  organizationId,
  params: req.params,
  query: req.query,
  payload: req.body,
})
```

For simple existing services, a controller may continue existing parameter names when the values are already parsed and sanitized:

```js
service.createThing({ organizationId, payload: req.body })
```

A controller must not pass the entire `req` object to a service.

## Error Contract Standard

BE-06 request validation errors should use a stable request validation error code:

```text
REQUEST_VALIDATION_ERROR
```

Validation error details should preserve useful field information where available:

```text
path
code
message
```

Module-local validation code must not invent unrelated request validation error codes unless an atomic task explicitly documents the reason.

## Request Validation vs Domain Validation

Request validators own:

```text
body shape
params shape
query shape
type coercion
required request fields
basic enum membership for request contracts
DTO construction / sanitized request payload
```

Services or policies own:

```text
domain invariants
lifecycle transition rules
authorization-sensitive validation
existence checks
cross-record consistency
database-backed validation
business outcome decisions
```

Validators must not import Prisma.

Validators must not perform database queries.

Validators must not silently change business behavior.

## Naming and Placement Standard

Preferred module validation file:

```text
fieldops-be/src/modules/<module>/<module>.validation.js
```

Allowed alternatives when existing repository convention requires them:

```text
<module>.validator.js
<module>.schema.js
<module>.dto.js
```

Each module validation file should export middleware or schema objects clearly named by operation.

Examples:

```js
validateCreateIssue
validateUpdateIssue
createIssueBodySchema
issueIdParamsSchema
listIssueQuerySchema
```

## Adoption Strategy

BE-06 must proceed incrementally.

Recommended order:

1. Keep `src/core/validation` as foundation.
2. Normalize one module at a time.
3. Prefer Issue as the first module because it currently has no route validation middleware.
4. Preserve Equipment behavior unless specifically normalizing it under BE-06.06.
5. Do not alter Prisma schema, migrations, app/server bootstrap, package files, or frontend files.

## BE-06.01 Definition of Done

BE-06.01 is complete when:

- Repository validation foundations are identified.
- One request validation middleware authority is selected.
- DTO handoff expectations are documented.
- Request validation and domain validation ownership are separated.
- Error code standard is documented.
- Allowed and forbidden implementation boundaries are clear.

## Review Checklist

Before implementation tasks continue, confirm:

- Does the target module use route-level validation?
- Are body, params, and query schemas explicit where needed?
- Does the service receive sanitized data?
- Are validation errors returned under `REQUEST_VALIDATION_ERROR` unless explicitly justified?
- Are domain lifecycle or database-backed checks kept out of request validators?
- Are changed files within BE-06 allowed scope?
