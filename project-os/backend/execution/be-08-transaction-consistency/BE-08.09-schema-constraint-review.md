## EquipmentCategory Schema Truth

Model:

```text
EquipmentCategory
```

Relevant fields:

```text
id             String @id @default(cuid())
organizationId String @map("organization_id")
code           String
name           String
status         CategoryStatus @default(ACTIVE)
```

Relevant constraints:

```text
@@unique([organizationId, code])
@@unique([organizationId, name])
@@index([organizationId])
@@map("equipment_categories")
```

BE-08 interpretation:

```text
Category code and category name are unique only inside the organization scope.
This supports organization-scoped conflict handling for category auto-create.
Duplicate category creation should map to a scoped conflict or retry-safe lookup strategy.
```

## Equipment Schema Truth

Model:

```text
Equipment
```

Relevant fields:

```text
id             String @id @default(cuid())
organizationId String @map("organization_id")
code           String
qrCode         String @map("qr_code")
name           String
categoryId     String? @map("category_id")
status         EquipmentStatus @default(READY)
condition      EquipmentCondition @default(GOOD)
retiredAt      DateTime? @map("retired_at")
deletedAt      DateTime? @map("deleted_at")
```

Relevant constraints:

```text
@@unique([organizationId, code])
@@unique([organizationId, qrCode])
@@index([organizationId])
@@index([categoryId])
@@index([status])
@@index([condition])
@@index([retiredAt])
@@index([deletedAt])
@@index([qrCode])
@@map("equipments")
```

BE-08 interpretation:

```text
Equipment code and qrCode are unique only inside the organization scope.
This supports duplicate-create conflict detection and retry-safety review.
Soft delete does not currently remove unique-key pressure because deletedAt is not part of the unique constraints.
Repeated create with the same code or qrCode should not create another equipment record in the same organization.
```

## Transaction Candidate Decision

### createEquipment with category auto-create

Current flow:

```text
find category by code
create category if missing
create equipment
```

Schema-backed risk:

```text
EquipmentCategory has unique scoped constraints on organizationId + code and organizationId + name.
Equipment has unique scoped constraints on organizationId + code and organizationId + qrCode.
Concurrent category auto-create can hit category unique conflict.
Concurrent equipment create can hit equipment unique conflict.
```

BE-08 decision:

```text
Transaction Boundary: REQUIRED
Idempotency Review: REQUIRED_DEFERRED
Conflict Review: REQUIRED
Rollback Review: REQUIRED
```

Reason:

```text
The operation may create a dependent category and then create equipment.
If equipment creation fails after category creation, the operation can leave a category created by a failed equipment intent unless a transaction or accepted orphan-category policy exists.
```

Preferred implementation direction:

```text
Use Unit-of-Work / Prisma transaction for category auto-create + equipment create.
Use scoped uniqueness to map duplicate category/equipment create to ConflictError or documented retry-safe behavior.
Do not modify schema in BE-08 unless explicitly assigned.
```

## Soft Delete Decision

Current schema:

```text

```

BE-08 decision:

```text
Idempotency Review: REVIEW

```

Reason:

```text

Unique constraints remain active for soft-deleted records.
```
