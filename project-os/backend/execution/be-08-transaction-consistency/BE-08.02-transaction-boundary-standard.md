# BE-08.02 Transaction Boundary Standard

Status: Draft
Phase: BE-08 Transaction and Consistency
Milestone: BE-08.02
Owner: BE-08 Transaction and Consistency Owner

## Purpose

This document defines the Backend transaction boundary standard for FieldOps.

The goal is to make state-changing service behavior safe, reviewable, organization-scoped, and consistent without moving domain business logic into shared foundation.

## Boundary Principle

A transaction boundary belongs where a backend operation must make multiple persistent changes succeed or fail as one unit.

```text
Validation
  -> Policy / scope checks
  -> Transaction boundary if required
  -> Persistent writes
  -> Response mapping
```

Transactions are execution safety tools. They do not own business meaning.

## Ownership Rule

### Route

Routes own only URL path, HTTP method, middleware chain, and auth / organization guard composition.

Routes must not open transactions.

### Controller

Controllers own request and response orchestration only.

Controllers must not open transactions unless there is no service layer and the exception is documented.

### Service

Services own transaction decisions.

A service must decide whether a use case requires a transaction after validation, scope checks, and policy checks are clear.

### Repository

Repositories execute persistence operations and may accept an injected transaction client.

Repositories must not secretly create transaction scope for business use cases unless the repository method itself is explicitly a low-level infrastructure helper.

### Shared Foundation

Shared transaction helpers may provide domain-neutral execution mechanics only.

Shared must not contain lifecycle decisions, inventory decisions, billing calculations, approval rules, closure rules, or response mapping.

## Required Transaction Rule

A service operation requires an explicit transaction when any of these are true:

```text
1. The operation writes to more than one persistent record or table.
2. The operation performs read -> validate current state -> write where stale state can cause an invalid transition.
3. The operation creates or updates a dependent record before completing the main mutation.
4. The operation combines domain mutation with audit, outbox, ledger, or module side-effect intent.
5. The operation must preserve inventory, assignment, billing, closure, approval, or lifecycle consistency.
6. The operation must be retry-safe or idempotent and needs a persistent idempotency record.
7. The operation must fail completely when any sub-step fails.
```

## Optional Transaction Rule

A service operation may skip an explicit transaction when all are true:

```text
1. It performs a single independent write.
2. Validation happens before mutation.
3. Organization scope is enforced.
4. No dependent record, side-effect intent, or status transition is coupled to the write.
5. Failure of the write leaves no partial business outcome.
```

## Forbidden Transaction Pattern

The following patterns are forbidden:

```text
controller-level business transactions
shared helper owning business branching
repository opening hidden business transactions
unscoped writes inside transaction work
validation after mutation
side-effect intent recorded outside the chosen consistency boundary
silently swallowing partial failure
opening nested transactions without an explicit rule
```

## Unit-of-Work Client Rule

Repository methods that may participate in transactions should accept an optional client parameter.

Preferred shape:

```text
repositoryMethod({ organizationId, ..., client })
```

or for existing repository style:

```text
repositoryMethod(organizationId, id, data, { client })
```

The service owns passing the transaction client.

## Nested Transaction Rule

Nested transactions are not allowed by default.

If a service calls another service inside a transaction, the called service must either accept the existing transaction context or be refactored into a domain operation that does not open its own transaction.

If this cannot be guaranteed, the operation must be marked `TRANSACTION_REVIEW_REQUIRED`.

## Side-Effect Intent Rule

Side-effect intent must be consistent with the domain mutation that creates it.

Preferred pattern:

```text
inside transaction: write domain record + write audit/outbox/intent record
outside transaction: dispatch from persisted intent after successful commit
```

If no outbox or intent table exists, document the deferral or risk.

## Organization Scope Rule

Organization scope must be established before mutation and preserved inside transaction work.

Minimum rule:

```text
all read-before-write checks include organizationId where applicable
all created records include organizationId where applicable
all updates/deletes verify scoped ownership before mutation when Prisma unique where cannot include organizationId
```

## Error and Rollback Rule

When a transaction fails:

```text
all persistent sub-writes must rollback
error mapping must remain intentional
partial success must not be returned
conflict and validation errors must keep backend error contract
unexpected errors must not expose internal persistence details
```

## Idempotency and Retry Rule

A mutation needs idempotency review when:

```text
client retry can create duplicate records
operation result may be unclear to caller after connection interruption
operation records side-effect intent
operation is billing, payout, notification, lifecycle-closing, or inventory-changing
```

If idempotency is not implemented, the operation must document one of:

```text
NOT_REQUIRED
REQUIRED_DEFERRED
BLOCKED_BY_SCHEMA
BLOCKED_BY_PRODUCT_CONTRACT
```

## Conflict Rule

A mutation needs conflict review when:

```text
two users can update the same record concurrently
status transitions depend on current state
inventory availability can be consumed by competing requests
approval or closure can be submitted twice
billing or payout can be finalized twice
```

Conflict behavior must be explicit:

```text
return ConflictError / 409
or use schema-supported optimistic concurrency
or serialize through transaction strategy
or document deferral
```

## BE-08.01 Candidate Application

### Equipment create with category auto-create

```text
Transaction candidate: YES
Risk: MEDIUM
Reason: category lookup/create can be followed by equipment create as a dependent write chain.
```

Expected BE-08 follow-up:

```text
Confirm schema constraints for category uniqueness.
Decide between explicit transaction, upsert-like category behavior, or documented acceptable risk.
```

### Equipment update with categoryCode/category reassignment

```text
Transaction candidate: REVIEW
Risk: MEDIUM
Reason: category lookup/create can occur before equipment update.
```

### Equipment soft delete

```text
Transaction candidate: NOT YET REQUIRED / REVIEW WHEN RELATIONS EXIST
Risk: LOW now, potentially MEDIUM/HIGH with active session, movement, checkout, or required-tool relations.
```

## Verification Requirements

Every transaction implementation must provide at least one of:

```text
unit test proving rollback
service test proving scoped mutation behavior
integration test proving conflict/idempotency behavior
documented deferral with reason and owner
```

Source inspection alone is acceptable only for standard documentation milestones, not for final implementation completion.
