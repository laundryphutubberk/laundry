# BE-08.04 Idempotency and Retry Safety

Status: Draft
Phase: BE-08 Transaction and Consistency
Milestone: BE-08.04
Owner: BE-08 Transaction and Consistency Owner

## Purpose

This document defines Backend idempotency and retry safety expectations for FieldOps.

The goal is to prevent duplicate persistent outcomes, unsafe repeated mutations, and unclear retry behavior for state-changing API operations.

## Core Principle

A retry-safe mutation is one where repeating the same client intent does not create a second unintended business outcome.

```text
same intent
same scope
same operation
same safe outcome
```

Idempotency is part of execution truth. It must be explicit for operations where retry can duplicate or corrupt state.

## Ownership Rule

### Frontend

Frontend may send a client request key or action key when a mutation requires retry safety.

Frontend does not decide backend idempotency semantics.

### Backend Service

Backend service owns the idempotency decision for a use case.

The service must decide whether retry safety is:

```text
NOT_REQUIRED
REQUIRED
REQUIRED_DEFERRED
BLOCKED_BY_SCHEMA
BLOCKED_BY_PRODUCT_CONTRACT
```

### Repository

Repository may store or read idempotency records when the service explicitly requests it.

Repository must not infer business idempotency by itself.

### Prisma / Schema

Persistent uniqueness, idempotency tables, version fields, and constraints belong to Prisma/schema truth.

Backend must not assume these constraints exist without reading schema or receiving Prisma approval.

## Idempotency Review Trigger

A mutation requires idempotency review when any of these are true:

```text
1. Client retry can create duplicate records.
2. Network interruption can leave caller unsure whether mutation completed.
3. The operation writes more than one persistent record.
4. The operation records audit/outbox/side-effect intent.
5. The operation affects billing, expense, payout, inventory, lifecycle closure, approval, or notification intent.
6. The operation can be submitted twice by the same actor.
7. The operation is expected to be called from mobile or unreliable field network conditions.
```

## Default Classification

### NOT_REQUIRED

Use when:

```text
single independent update
safe overwrite semantics
no duplicate business outcome
no side-effect intent
no status transition risk
```

### REQUIRED

Use when:

```text
operation creates durable business record from client intent
operation must be retried safely
operation is multi-write and externally triggered
operation has side-effect intent
```

### REQUIRED_DEFERRED

Use when idempotency is needed but cannot be implemented in the current milestone.

The deferral must include:

```text
reason
owner
risk
expected follow-up phase or milestone
```

### BLOCKED_BY_SCHEMA

Use when safe implementation requires schema support such as:

```text
unique idempotency key
version field
operation key table
unique composite constraint
outbox table
```

### BLOCKED_BY_PRODUCT_CONTRACT

Use when product behavior is unclear, such as:

```text
should duplicate submission return old result or conflict
how long an idempotency key lives
which actor/scope owns the key
what response should be replayed
```

## Idempotency Key Scope

An idempotency key must never be global by accident.

Minimum recommended scope:

```text
organizationId
actorId when available
operationName
idempotencyKey
```

For module-specific operations, add the target or parent resource when needed:

```text
fieldSessionId
issueId
equipmentId
billingPeriodId
```

## Retry Response Rule

When the same idempotency key is reused for the same operation and same scope, Backend should choose one explicit response behavior:

```text
REPLAY_PREVIOUS_RESULT
RETURN_CONFLICT
RETURN_CURRENT_RESOURCE_STATE
```

The behavior must be documented per operation.

## Conflict Rule

If the same idempotency key is reused with a different payload inside the same scope, the operation must not silently proceed.

Preferred behavior:

```text
409 Conflict
code: IDEMPOTENCY_KEY_CONFLICT
```

Exact error shape must follow the backend error contract of the project.

## Retry Safety Without Idempotency Table

Some operations may be retry-safe through existing persistent constraints.

Examples:

```text
unique natural key prevents duplicate create
upsert-like behavior is intentional and tested
status transition is guarded by current state
operation returns existing scoped resource when duplicate intent is detected
```

This must be documented as `NOT_REQUIRED_WITH_CONSTRAINT` or equivalent in the operation notes.

## Transaction Relationship

Idempotency and transaction boundaries must work together.

Recommended flow for REQUIRED idempotency:

```text
validate request
establish organization scope
check or reserve idempotency record
run transaction-bound mutation
record result or completion state
return documented response
```

If idempotency state and domain mutation must stay consistent, they belong in the same transaction boundary.

## Equipment Candidate Application

### createEquipment

Current classification:

```text
Idempotency Review: REQUIRED_DEFERRED
Risk: duplicate equipment create on retry if natural uniqueness is not confirmed
Reason: code / qrCode uniqueness and behavior must be verified against Prisma schema truth
```

Expected next check:

```text
inspect Prisma schema constraints for equipment code, qrCode, organizationId
confirm desired product behavior for duplicate create retry
```

### updateEquipment

Current classification:

```text
Idempotency Review: NOT_REQUIRED or REVIEW
Reason: mostly update/overwrite semantics, but category auto-create path may require transaction/idempotency review
```

### updateEquipmentStatus

Current classification:

```text
Idempotency Review: REVIEW
Reason: status transition behavior may depend on current state and policy rules
```

### deleteEquipment

Current classification:

```text
Idempotency Review: REVIEW
Reason: soft delete may be retry-safe if repeated delete returns same deleted state, but current behavior must be verified
```


## Verification Requirements

For REQUIRED idempotency, provide one or more:

```text
same key same payload returns documented safe result
same key different payload returns conflict
retry after success does not duplicate records
retry after partial failure follows documented behavior
organization scope prevents key collision across organizations
```

For REQUIRED_DEFERRED, provide:

```text
reason for deferral
risk statement
owner for follow-up
blocked dependency
```

## Implementation Gate

Do not implement idempotency runtime changes until:

```text
schema support is confirmed
operation-level response behavior is approved
test strategy is available
transaction boundary relationship is clear
error contract is confirmed
```
