# BE-08.02 Transaction Boundary Standard

Status: Draft — requires BE-08.02 Gate Review before runtime enforcement  
Owner: Backend Architect  
Scope: Transaction, idempotency, side effects, and cross-boundary consistency

## 1. Objective

Define safe consistency rules for:

- local multi-write workflows inside the Laundry database
- retry-sensitive mutations
- external provider calls
- future cross-module or cross-process workflows

This standard must not replace existing Prisma transactions with Saga by default.

## 2. Boundary Selection Rule

Choose the smallest consistency mechanism that safely covers the workflow.

| Workflow Type | Required Mechanism |
|---|---|
| Multiple writes in one PostgreSQL database | Prisma interactive transaction |
| Single write protected by a unique constraint | Database constraint plus conflict mapping |
| Concurrent state transition | Optimistic guard, version check, or approved row-lock strategy |
| Retry-sensitive command | Natural idempotency key or explicit idempotency key |
| External side effect after database commit | Outbox/event or explicit post-commit handling |
| Cross-process workflow with independent commits | Saga/compensating actions after architecture approval |

## 3. Local Transaction Standard

For workflows writing multiple records in the same database:

1. Open one transaction at the service/application workflow boundary.
2. Pass the transaction-compatible client to every participating repository.
3. Commit all authoritative database writes together.
4. Roll back all authoritative database writes together on failure.
5. Do not perform slow network/provider calls while holding a database transaction open.

Repositories must accept the approved client parameter pattern and remain free of business-flow orchestration.

## 4. Idempotency Rule

Every retry-sensitive mutation must document one approved strategy:

- natural key plus unique database constraint
- explicit `idempotencyKey`
- state/version precondition
- deterministic command identity

An explicit `idempotencyKey` is mandatory only when natural uniqueness or state preconditions cannot safely prevent duplicate execution.

The implementation must define:

```text
Key scope
Key lifetime
Stored result or replay behavior
Conflict response
Retry behavior
Workspace ownership
```

Do not introduce a distributed lock or cache dependency without Technology/ADR approval.

## 5. Concurrency Rule

A method name such as `find...ForUpdate` must not imply locking unless the query actually acquires an approved database lock.

For state transitions, choose and document one of:

- conditional update using current state
- version field / optimistic concurrency
- database row lock
- unique constraint
- serialized workflow ownership

Lost-update risks must be verified with concurrent execution tests where practical.

## 6. Compensating Transaction Flow

Use Saga/compensating actions only when a workflow cannot be committed atomically in one database transaction.

When approved:

1. Commit the local authoritative state.
2. Record the next action durably when required.
3. Invoke the downstream operation.
4. On downstream failure, invoke the registered compensating action or move the workflow to an explicit recovery state.
5. Make forward and compensating actions idempotent.
6. Record correlation and execution state for recovery.

A compensating action must be defined in the owning domain contract. Compensation must not silently erase audit history.

## 7. Inventory Projection Rule

`LinenMovement` is the movement ledger/source of inventory truth.

The chosen projection strategy must be approved in BE-08 Gate Review:

### Synchronous projection

Movement and `LinenInventorySummary` update in the same database transaction.

Use when:

- both records share one database
- immediate consistency is required
- write volume remains manageable

### Asynchronous projection

Movement commits first; an outbox/job updates the summary later.

Use only after approving:

- outbox/event design
- replay/rebuild strategy
- duplicate-event handling
- lag monitoring
- recovery procedure

## 8. Side-Effect Matrix

| Side Effect | Default Position |
|---|---|
| Authoritative domain writes | Inside transaction |
| Status/audit record required for consistency | Inside transaction |
| Derived summary in synchronous mode | Inside transaction |
| Runtime log | Outside transaction; correlated with request/transaction |
| Notification | After commit or through outbox |
| External provider call | Outside transaction unless explicitly justified |
| Cache invalidation | After commit; retry-safe |

## 9. Anti-Corruption Layer for External APIs

External integrations must be wrapped behind a provider/ACL boundary.

The boundary must define:

- timeout
- retry policy
- circuit-breaker policy when justified
- error mapping
- correlation/request id propagation
- idempotency behavior
- sensitive-data handling

Timeouts must be explicitly configured per provider. A 5000 ms limit may be used as an initial ceiling only when verified appropriate; it is not a universal constant.

## 10. Evidence and Verification

A transaction capability may freeze only with evidence appropriate to its risk:

```text
Static dependency verification
Repository transaction-client verification
Runtime workflow verification
Rollback/failure-path verification
Duplicate-request verification
Concurrency verification
Inventory consistency verification
Regression verification
```

Freeze documents must describe observed implementation truth. Planned Saga, metrics, locking, or idempotency capabilities must be marked `NOT IMPLEMENTED` until verified.

## 11. Stop Conditions

Stop and request Gate/ADR review if implementation requires:

- new distributed lock/cache infrastructure
- asynchronous outbox or queue
- cross-service Saga orchestration
- schema/version field changes
- aggregate-boundary changes
- inventory source-of-truth changes
- API contract changes
- workspace or permission model changes

## 12. Current Laundry Position

Current verified active Work/Bag flows use Prisma transactions and transaction-compatible repositories.

Movement runtime, Inventory Summary projection, generalized idempotency storage, and cross-process Saga orchestration must remain gap/proposal items until implemented and verified.
