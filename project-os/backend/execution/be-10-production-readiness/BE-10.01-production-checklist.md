# BE-10.01 — Production Readiness Playbook

Status: Standard
Scope: Backend Production Readiness
Owner: Backend Architecture
Parent Package: `BE-10-production-readiness.md`

## Purpose

BE-10.01 defines the production readiness playbook for a backend release scope.

The goal is not only to provide a checklist. The goal is to decide whether a backend change has enough architecture, contract, safety, observability, and verification evidence to continue toward deployment review.

## When To Use

Use BE-10.01 before a backend release scope moves into deployment planning.

Use it for:

- backend API behavior changes
- service or repository behavior changes
- validation or policy changes
- transaction, consistency, or audit changes
- runtime foundation changes
- documentation-only release gates when production readiness needs to be recorded

## Inputs

Collect these inputs before reviewing readiness:

```text
Release scope
Target modules
Changed files
Related commits or pull request
Known backend contracts affected
Known data or migration impact
Available verification evidence
Known exceptions or deferrals
```

## Release Scope Record

For a real release, replace the placeholders below.

```text
Release scope: required
Target modules: required
Target environment: required when known
Related commits: required
Related PR: required when available
Reviewer: required
```

## Review Procedure

Run the review in this order:

1. Identify the release scope.
2. List changed files.
3. Classify the change type.
4. Check architecture boundaries.
5. Check request, response, and error contracts.
6. Check safety and organization scope.
7. Check observability and evidence quality.
8. Record verification result.
9. Record exceptions.
10. Decide readiness status.

Do not mark the release scope ready only because tests pass. Tests are evidence, not the full production readiness decision.

## Change Classification

Classify the release scope with one or more values:

```text
DOC_ONLY
TEST_ONLY
API_CONTRACT_CHANGE
SERVICE_BEHAVIOR_CHANGE
REPOSITORY_BEHAVIOR_CHANGE
VALIDATION_CHANGE
POLICY_CHANGE
TRANSACTION_OR_CONSISTENCY_CHANGE
RUNTIME_FOUNDATION_CHANGE
CONFIGURATION_CHANGE
MIGRATION_OR_DATA_CHANGE
```

The classification determines which checklist sections are mandatory.

## Architecture Review

A backend change is acceptable when ownership boundaries remain clear.

Mandatory checks:

```text
□ route/controller/service boundaries are preserved
□ service ownership remains clear
□ shared foundation does not absorb domain behavior
□ repository behavior is not changed outside the release scope
□ no unrelated module refactor is included
□ no unapproved schema change is included
□ repository reality is documented when structure changes
```

Fail this section if a change mixes unrelated architecture concerns or moves business meaning into the wrong layer.

## Contract Review

A backend change is acceptable when external behavior is intentional and recorded.

Mandatory checks:

```text
□ request shape is known for changed endpoints
□ response shape is known for changed endpoints
□ error shape is intentional and consistent
□ authorization expectation is known
□ organization scope is enforced where required
□ frontend-visible behavior is documented when affected
```

If no API contract is affected, record `NOT_APPLICABLE` with a reason.

## Safety Review

A backend change is acceptable when it does not introduce uncontrolled production risk.

Mandatory checks:

```text
□ validation occurs before mutation
□ scoped operations preserve organization or tenant boundaries
□ sensitive data is not exposed in responses, logs, or errors
□ state-changing operations have transaction or consistency expectations reviewed
□ release risk is classified as LOW, MEDIUM, or HIGH
□ known exceptions are documented
```

Risk classification:

```text
LOW: documentation, tests, or isolated non-runtime behavior
MEDIUM: service behavior, validation, policy, or API behavior change
HIGH: migration, data mutation, transaction, auth, organization scope, or runtime foundation change
```

## Observability Review

A backend change is acceptable when failures can be diagnosed safely.

Mandatory checks:

```text
□ logs provide safe diagnostic context when runtime behavior changes
□ request id or correlation context is preserved where available
□ audit behavior is separated from normal logging when required
□ metrics or traces are defined when required
□ smoke or verification evidence can be reviewed by the next steward
```

For documentation-only or test-only changes, observability may be `NOT_APPLICABLE` with a reason.

## Verification Review

A backend change is acceptable when evidence is reproducible.

Evidence types:

```text
AUTOMATED_TEST
BUILD_OR_STATIC_CHECK
SOURCE_INSPECTION
LOCAL_RUNTIME_EVIDENCE
CI_EVIDENCE
MANUAL_DEFERRAL
```

Mandatory checks:

```text
□ automated tests pass, or deferral is documented
□ build or static verification passes, or deferral is documented
□ manual source inspection is recorded when runtime execution is unavailable
□ smoke check is defined or completed when release scope requires it
□ verification result is reproducible by the next steward
```

## Evidence Quality Levels

Use one evidence quality level:

```text
STRONG: CI or local automated verification plus clear changed-file scope
ACCEPTABLE: source inspection plus documented reasoning
LIMITED: partial evidence with clear deferrals
INSUFFICIENT: missing or non-reproducible evidence
```

## Decision Rules

Use one result only:

```text
READY
READY_WITH_NOTES
NOT_READY
```

Decision rules:

```text
READY: all mandatory checks pass and no blocking exceptions remain.
READY_WITH_NOTES: mandatory checks pass with accepted non-blocking exceptions or deferrals.
NOT_READY: one or more mandatory checks fail, evidence is insufficient, or blocking exceptions remain.
```

## Known Exceptions

Exceptions must be explicit. Do not leave `TBD` in a completed review.

Each exception must include:

```text
Exception
Impact
Blocking: yes/no
Owner
Next action
```

## Required Return

A completed BE-10.01 return must include:

```text
Commit List
Files Changed
Change Classification
Architecture Result
Contract Result
Safety Result
Observability Result
Verification Result
Evidence Quality
Known Exceptions
Production Readiness Result
Next Step
```

## Handoff

If BE-10.01 returns `READY` or `READY_WITH_NOTES`, continue to:

```text
BE-10.02 Deployment Checklist
```

If BE-10.01 returns `NOT_READY`, resolve the blocking issue before deployment planning.
