## Purpose

Normalize the Issue module so organization-scope business enforcement belongs to the service/policy layer rather than the HTTP controller.

This keeps controllers focused on HTTP request/response orchestration and moves Issue organization boundary decisions into the business layer.

## Business Workflow Before

The Issue controller derived `organizationId` from the request and also asserted that organization context existed before calling the service.

The service executed issue use cases and delegated persistence to the repository, but organization context enforcement was partly owned by the controller.

## Business Workflow After

The Issue controller now:

- derives request context from `req`;
- calls the matching issue service method;
- maps response DTOs;
- returns HTTP success responses;
- passes errors to `next`.

The Issue service now:

- asserts Issue organization context through policy before every scoped issue workflow;
- owns the business boundary for list, queue, read, equipment lookup, create, update, and delete/resolve flows;
- delegates data access to the repository;
- preserves existing response and error behavior.

The Issue policy now exposes:

```text
assertIssueOrganizationContext
assertIssueExists
```

## API Contract

No intentional API contract change.

Existing routes, response mappers, status codes, and exported controller method names were preserved.

The same `IssueOrganizationContextRequiredError` is still used when organization context is missing.

## Repository Boundary

No repository source was changed.

The repository remains data-access only for this milestone.

## Validation Boundary

`issue.validator.js` was inspected but not changed.

Validation remains existing source reality. Broader validation redesign belongs to BE-06.

## Policy Boundary

Organization context enforcement now sits in `issue.policy.js` and is called from `issue.service.js`.

This aligns with BE-05 business-layer ownership and leaves future reusable domain-policy expansion to BE-07 if needed.

## Transaction Boundary

No transaction was introduced.

The update and delete/resolve flows still first verify issue existence inside organization scope and then perform the mutation by issue id.

If stronger atomicity is required, BE-08 should review whether existence check and mutation should be wrapped in a transaction or represented by organization-scoped mutation methods.
