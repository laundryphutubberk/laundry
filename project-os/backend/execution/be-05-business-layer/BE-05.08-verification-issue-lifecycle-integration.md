## Repository Reality

Current verification-related behavior is represented by Field Session lifecycle and scan workflow.

The active Field Session workflow owns:

- scan mode resolution
- session status compatibility for scans
- session item status transition
- equipment movement creation
- close readiness based on unresolved item statuses

The active Issue workflow owns:

- issue listing
- issue queue
- issue lookup by id
- open issue lookup by equipment id
- issue creation
- issue update
- issue resolve/delete behavior

There is currently no active service that orchestrates verification-to-issue creation or issue-to-verification closure.

## Business Boundary Decision

BE-05.08 does not create a new verification route/module because route/API surface ownership belongs to BE-03 and validation/API contract definition belongs to later execution packages unless explicitly routed.

For the active branch, the normalized integration boundary is:

```text
Field Session service
  owns operational scan and verification-adjacent lifecycle transitions

Issue service
  owns issue lifecycle and open issue queue behavior

Future Verification service
  should own verification workflow orchestration when introduced
  should call Issue service/repository boundaries through an explicit contract
  should not hide issue lifecycle decisions inside controller or repository files
```

## Intended Future Integration Contract

When a verification backend module is introduced, the service boundary should be shaped as use cases such as:

```text
startVerificationForSession
completeVerificationForSession
recordVerificationIssue
resolveVerificationIssue
getVerificationReadiness
```

The future service should coordinate:

- Field Session lifecycle state
- session item status
- equipment movement evidence
- Issue lifecycle creation/update/resolution
- deferred audit intent for BE-09

The repository layer should only expose data-access methods required by these workflows.

## Issue Integration Boundary

Issue service currently remains independent and organization-scoped.

Any future verification workflow that creates or resolves issues should preserve Issue service ownership or define a clear orchestration service that coordinates issue behavior without moving issue decisions into a controller or repository.

## Field Session Integration Boundary

Field Session scan workflow currently creates equipment movement records and controls status transition eligibility.

Verification should not bypass these lifecycle guards.

Future verification implementation should reuse or explicitly coordinate with:

```text
fieldSessionLifecycle.policy.js
fieldSessionLifecycle.guard.js
fieldSession.service.js
```
