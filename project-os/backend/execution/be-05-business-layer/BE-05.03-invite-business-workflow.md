## Purpose

Normalize the Invite module business workflow by making invite context boundaries explicit in policy and keeping service orchestration responsible for workflow decisions.

This milestone preserves the current HTTP/API contract while clarifying service and policy ownership.

## Business Workflow Before

The active Invite controller already delegated to the service layer and remained mostly HTTP-focused.

The service owned invite workflows including:

- invite listing;
- invite creation;
- invite code resolution;
- invite join transaction;
- invite revoke workflow.

However, user and organization context checks were implemented as service-local helpers that directly constructed context errors.

## Business Workflow After

The Invite policy now exposes explicit context boundary helpers:

```text
assertInviteOrganizationContext
assertInviteUserContext
```

The Invite service now uses those policy helpers through `requireOrganizationId` and `requireUserId`.

This preserves service orchestration while moving reusable business boundary assertions into the policy layer.

## API Contract

No intentional API contract change.

Routes, controller exports, response mappers, service method names, and repository method names were preserved.

## Repository Boundary

No repository source was changed.

The repository remains responsible for data access only, including invite lookup, membership lookup, invite mutation, and transaction wrapper access.

## Transaction Boundary

`joinInvite` still runs inside `inviteRepository.transaction`.

This transaction coordinates invite lookup, expiry handling, active user lookup, membership creation, and invite acceptance.

BE-08 should later review whether this transaction boundary is sufficient for invite acceptance consistency and duplicate join protection under concurrency.

## Validation Boundary

`invite.validator.js` was not changed.

Request validation remains out of scope for BE-05 except for existing source reality. Broader validation belongs to BE-06.

## Policy Boundary

Invite policy now owns:

- organization context presence;
- user context presence;
- invite existence;
- invite pending state;
- invite expiration semantics;
- organization availability;
- user availability.

## Audit / Observability Boundary

No audit behavior was introduced.

Invite lifecycle events may need audit intent in BE-09, especially create, accept, expire, and revoke.
