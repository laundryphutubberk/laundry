## Findings

### Logger foundation

- Logger exports are centralized through `src/core/logger/index.js`.
- Logger writes structured JSON entries with `level`, `message`, `timestamp`, and normalized metadata.
- Supported levels are `info`, `warn`, and `error`.
- Logger writes to `console.info`, `console.warn`, or `console.error` based on level.

### Request logging

- Runtime bootstrapping installs `requestId()` before `requestLogger`.
- Request logger records `request.started` and `request.finished` events.
- Request logs include request id, method, path, status code, and duration where available.
- Request id is sourced from `req.id`, `req.requestId`, or `x-request-id`.

### Request id propagation

- Request id middleware accepts incoming `x-request-id` or generates a new id.
- Request id is assigned to `req.id` and `req.requestId`.
- Request id is returned on the response header.
- Request context stores request id, method, and path for downstream investigation support.

### Audit boundary note

- `audit.service.js` currently emits audit lifecycle messages through the logger.
- This is acceptable for the current lightweight implementation but should be reviewed again during `BE-09.04 Audit Alignment` because BE-09 requires audit behavior to be separated from normal logging when audit service behavior is required.
