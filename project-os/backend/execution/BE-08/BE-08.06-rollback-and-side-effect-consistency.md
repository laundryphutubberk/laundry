# BE-08.06 Rollback and Side-Effect Consistency

Status: Draft
Phase: BE-08 Transaction and Consistency
Milestone: BE-08.06
Owner: BE-08 Transaction and Consistency Owner

## Purpose

This document defines rollback and side-effect consistency expectations for FieldOps Backend.

The goal is to ensure that a state-changing operation does not leave persistent data, audit intent, outbox intent, or module side-effect records in a misleading partial state.

## Core Principle

A backend operation must not report success unless the required persistent outcome is complete.

```text
all required writes succeed
  -> operation succeeds
any required write fails
  -> operation fails and required writes rollback
```

Side effects must be aligned with the durable state that caused them.

## Ownership Rule

### Service

Service owns the business meaning of rollback and side-effect intent.

A service must define:

```text
which writes are required
which side-effect intents are part of the same outcome
which external actions must wait until commit
what failure means to the caller
```

### Repository

Repository executes persistence operations and supports transaction client usage.

Repository must not record side-effect intent unless the service explicitly requests it as part of the use case.

### Shared Foundation

Shared helpers may provide transaction mechanics, outbox mechanics, or audit recording mechanics only when domain-neutral.

Shared must not decide whether a field session is closed, equipment is returned, billing is final, or an expense is approved.

## Rollback Required Rule

Rollback consistency is required when an operation:

```text
1. writes more than one persistent record
2. creates a dependent record before the main record
3. updates parent and child records together
4. records audit/outbox/side-effect intent with domain mutation
5. changes inventory, billing, payout, approval, closure, or lifecycle state
6. must not leave partial user-visible outcome
```

## Side-Effect Intent Rule

If a side effect depends on a domain mutation, the intent should be persisted inside the same consistency boundary as the mutation.

Preferred pattern:

```text
transaction begins
  -> domain mutation
  -> audit/outbox/intent record
transaction commits
  -> async dispatcher processes persisted intent
```

If the dispatcher fails after commit, the persisted intent remains retryable.

If the transaction fails, the side-effect intent rolls back with the domain mutation.

## External Action Rule

External actions should not run inside the transaction callback.

Examples:

```text
sending notification
sending email
calling webhook
charging payment provider
removing object storage file
calling external audit sink
```

Preferred behavior:

```text
write intent inside transaction
perform external action after commit from persisted intent
record result separately
```

If this pattern is not available, document the risk and owner.

## Audit Consistency Rule

Audit records that prove the domain mutation occurred should be written in the same transaction when they are part of the required completion evidence.

Audit records that describe attempted actions may be recorded outside the transaction only if clearly marked as attempt-level evidence.

Required distinction:

```text
MUTATION_COMMITTED audit
MUTATION_ATTEMPTED audit
MUTATION_FAILED audit
```

## Outbox Consistency Rule

Outbox or notification intent is preferred for side effects that must happen after successful mutation.

Minimum outbox intent should include:

```text
organizationId
operation name
target resource type
target resource id
payload or payload reference
status
createdAt
retry metadata when available
```

Exact schema belongs to Prisma/schema truth and must not be assumed by BE-08 documentation alone.

## Partial Failure Rule

Partial success must not be returned unless the operation contract explicitly supports partial results.

Default response behavior:

```text
required sub-step fails -> operation fails
all required writes rollback -> no partial success returned
optional post-commit side effect fails -> mutation may remain committed, side-effect intent remains retryable or failed with evidence
```

## Equipment Candidate Application

### createEquipment with category auto-create

Rollback review:

```text
REQUIRED
```

Reason:

```text
When category creation and equipment creation happen in one user intent, equipment creation failure after category creation can leave an unused category.
```

Expected follow-up:

```text
verify schema constraints
choose transaction boundary or acceptable documented orphan-category behavior
add test if implemented
```

### updateEquipment category reassignment

Rollback review:

```text
REVIEW
```

Reason:

```text
Category auto-create before equipment update can leave unused category if the equipment update fails.
```

### updateEquipmentStatus

Rollback review:

```text
REVIEW NOW / REQUIRED WHEN AUDIT OR MOVEMENT INTENT EXISTS
```

Reason:

```text
Status update alone may be single-write today, but becomes rollback-sensitive when audit, movement, checkout, or return intent is added.
```

### deleteEquipment soft delete

Rollback review:

```text
REVIEW NOW / REQUIRED WHEN RELATED RECORDS EXIST
```

Reason:

```text
Soft delete alone is single-write today, but future session, movement, or return-verification links may require coordinated changes.
```

## Workflow Module Defaults

The following planned workflow modules should require rollback and side-effect consistency review before completion:

```text
field-session
issue
attachment
site-exit-verification
return-verification
billing
expense
notification
```

High-risk operations include:

```text
field-session close
site-exit verification submit
return verification finalize
billing finalize
expense approve or reject
payout create or mark paid
notification intent creation
attachment metadata plus storage reference update
```

## Verification Requirements

Rollback-sensitive operations must provide one or more:

```text
multi-write success test
second write failure rolls back first write
side-effect intent rolls back when domain mutation fails
post-commit dispatcher failure does not undo committed domain mutation but leaves retry evidence
organization scope remains preserved throughout rollback path
```

Documentation-only milestones may use source inspection and risk documentation, but implementation completion requires test evidence or explicit deferral.

## Deferral Format

If rollback or side-effect consistency is deferred, record:

```text
operation
required consistency behavior
risk if not implemented
blocking dependency
owner
next milestone
```

## Implementation Gate

Do not implement rollback or side-effect runtime changes until:

```text
transaction boundary is approved
schema/outbox/audit support is verified
operation-level side-effect contract is known
test strategy is available
error contract is confirmed
```
