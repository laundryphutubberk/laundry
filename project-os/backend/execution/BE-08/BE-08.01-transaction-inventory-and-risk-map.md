## Inventory Classification

### Confirmed Runtime Mutation Candidate: Equipment

Module:

```text
fieldops-be/src/modules/equipment/
```

Observed service mutation operations:

```text
createEquipment
updateEquipment
updateEquipmentStatus
deleteEquipment
findOrCreateCategoryByCode
```

Observed repository mutation operations:

```text
createRuntimeEquipment
updateRuntimeEquipment
deleteRuntimeEquipment
createRuntimeCategory
```

Risk classification:

```text
Risk: MEDIUM
Reason: createEquipment may create a category before creating equipment. This is a multi-write flow when categoryCode/category is supplied and the category does not already exist.
```

Organization scope requirement:

```text
Required. Equipment and category lookups include organizationId in runtime reads. Category creation and equipment creation include organizationId in data.
```

Validation prerequisite:

```text
Required fields: code, qrCode, name.
Status and condition are normalized before mutation.
Manual status updates are policy-gated before mutation.
```

Transaction observation:

```text
No explicit transaction boundary is currently visible in the equipment service/repository flow inspected.
The createEquipment path can execute category creation and equipment creation as separate writes.
```

BE-08 follow-up:

```text
Classify createEquipment(category auto-create + equipment create) as transaction-boundary candidate.
Decide whether category creation should be idempotent/upsert-like, unique-constrained, or wrapped in a unit of work.
```

## Risk Levels

```text
LOW
Single-record mutation, scoped, validated, no side effects, no dependent write.

MEDIUM
Multi-step read-before-write, soft delete, category/lookup creation, or status transition.

HIGH
Multiple persistent writes, inventory/billing/payout/lifecycle transition, external side effect, audit/outbox coupling, retry/concurrency exposure.
```

## Initial Transaction Boundary Candidates
