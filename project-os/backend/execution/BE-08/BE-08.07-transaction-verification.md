# BE-08.07 Transaction Verification

Status: Draft
Phase: BE-08 Transaction and Consistency
Milestone: BE-08.07
Owner: BE-08 Transaction and Consistency Owner

## Purpose

This document defines verification expectations for BE-08 transaction and consistency work.

The goal is to make transaction behavior provable through tests, source inspection, or explicit deferral evidence.

## Verification Principle

A transaction or consistency claim is not complete until it has evidence.

```text
implementation
  -> verification
  -> documented result
  -> handoff
```

Documentation-only milestones may use source inspection and risk classification. Runtime implementation milestones require test or build evidence unless explicitly deferred.

## Evidence Levels

### Level 1: Documentation Evidence

Acceptable for planning and standard-setting milestones.

Examples:

```text
source files read
risk map created
standard documented
candidate operations identified
deferrals recorded
```

### Level 2: Source Inspection Evidence

Acceptable when the task is to verify structure or confirm ownership.

Examples:

```text
service owns business decision
repository accepts client parameter
controller does not open transaction
organization scope is present before mutation
```

### Level 3: Unit Test Evidence

Required for helpers and small service rules when implementation changes runtime behavior.

Examples:

```text
runInTransaction returns callback result
runInTransaction passes transaction client
runInTransaction preserves thrown error
repository uses injected client
service throws conflict on stale state
```

### Level 4: Integration Test Evidence

Required for multi-write operation behavior when repository and database behavior matter.

Examples:

```text
first write rolls back when second write fails
same idempotency key does not duplicate records
unique constraint maps to expected conflict
scoped update cannot cross organization boundary
side-effect intent rolls back with domain mutation
```

## Required Verification Matrix

| BE-08 Area | Minimum Evidence | Strong Evidence |
|---|---|---|
| Transaction boundary | Source inspection + service test | Integration rollback test |
| Unit-of-Work helper | Unit test | Unit test + service adoption test |
| Idempotency | Deferral or service test | Integration duplicate retry test |
| Conflict handling | Service test | Conditional update / schema-backed test |
| Rollback | Integration test or deferral | Failure-in-middle rollback test |
| Side-effect consistency | Deferral or source inspection | Outbox/intent rollback test |
| Organization scope | Source inspection | Cross-organization negative test |

## Standard Test Scenarios

### Transaction Success

```text
Given valid scoped input
When the service executes a transaction-bound operation
Then all required records are created or updated
And the response maps the committed state
```

### Rollback on Failure

```text
Given a multi-write operation
When a later write fails
Then earlier writes are not committed
And no partial success is returned
```

### Idempotent Retry

```text
Given the same organization, actor, operation, and idempotency key
When the same request is submitted again
Then the backend returns the documented retry-safe outcome
And no duplicate business record is created
```

### Idempotency Conflict

```text
Given an idempotency key already used for one payload
When the same key is reused with a different payload
Then the backend returns the documented conflict response
```

### Stale State Conflict

```text
Given a resource state has changed after a caller read it
When the caller submits a transition based on stale state
Then the backend returns a conflict or documented safe outcome
```

### Organization Scope Preservation

```text
Given resources from two organizations
When a transaction-bound operation runs for one organization
Then reads, writes, conflicts, and idempotency keys remain scoped to that organization
```

### Side-Effect Intent Consistency

```text
Given a domain mutation records outbox or audit intent
When the domain mutation fails
Then the intent record is not committed
When the domain mutation succeeds
Then intent exists for post-commit processing
```

## Mocking Rules

Unit tests may mock Prisma transaction behavior when testing helper mechanics.

Service tests should prefer mocking repositories only when verifying service-owned branching.

Integration tests should use real persistence behavior when verifying rollback, unique constraint, or concurrency behavior.

## Deferral Requirements

When verification cannot be completed, the deferral must record:

```text
operation
expected behavior
missing evidence
reason verification is deferred
risk level
owner
next milestone
```

Allowed deferral reasons:

```text
schema support not confirmed
test harness missing
product contract missing
external side-effect infrastructure missing
module not implemented yet
```

Before workflow modules are marked complete, require BE-08 review for:

```text

issue escalation or resolution
site-exit verification submit
return verification finalize
billing finalize

```

Each must provide tests or documented deferral for:

```text

rollback
conflict behavior

side-effect intent consistency when applicable
organization scope
```
