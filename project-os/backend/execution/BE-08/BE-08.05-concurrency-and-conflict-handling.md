# BE-08.05 Concurrency and Conflict Handling

Status: Draft
Phase: BE-08 Transaction and Consistency
Milestone: BE-08.05
Owner: BE-08 Transaction and Consistency Owner

## Purpose

This document defines Backend concurrency and conflict handling expectations for FieldOps.

The goal is to make competing writes, stale state transitions, and double-submit situations explicit, testable, and safe.

## Core Principle

Concurrency risk exists when two valid requests can observe compatible state but only one should be allowed to complete the final business outcome.

```text
read current state
  -> decide allowed action
  -> write new state
```

If another request can change the same state between read and write, the operation needs conflict review.

## Ownership Rule

### Service

Service owns conflict decisions because conflict meaning is business execution meaning.

Service must decide:

```text
what state is required before mutation
what state change is allowed
what happens when state has already changed
what response is returned to caller
```

### Repository

Repository may support conflict checks through scoped reads, conditional updates, version checks, or unique constraints.

Repository must not invent conflict meaning without service direction.

### Prisma / Schema

Schema owns available persistence tools such as unique constraints, version columns, timestamps, and relational constraints.

Backend must not assume schema-level concurrency support without schema verification.

## Conflict Review Trigger

A mutation requires conflict review when any of these are true:

```text
1. Status transition depends on current state.
2. A record can be finalized, closed, approved, assigned, checked out, returned, paid, or cancelled.
3. Two users can update the same resource at the same time.
4. Inventory or equipment availability can be consumed by competing requests.
5. Duplicate submissions can create duplicate business outcomes.
6. A parent and child record must remain consistent during update.
7. Operation uses read-before-write to enforce policy.
```

## Standard Conflict Outcomes

Backend must choose one explicit conflict outcome per operation:

```text
409 Conflict
idempotent replay / current state return
safe no-op with documented response
retry allowed with updated state
manual review required
```

Silent overwrite is not allowed for stateful lifecycle operations.

## Recommended Error Shape

Preferred semantic error:

```text
status: 409
code: CONFLICT
errorCode: RESOURCE_STATE_CONFLICT or equivalent project standard
message: action-specific human-safe message
```

Exact response shape must follow the current backend error contract.

## Strategy Options

### Scoped Read Before Write

Use when current state must be verified before mutation.

Minimum rule:

```text
read includes organizationId where applicable
service checks allowed transition
write is scoped or guarded by prior verified identity
```

Risk:

```text
read-before-write alone can still be stale without additional guard
```

### Conditional Update

Use when operation must only proceed if current state still matches expectation.

Preferred idea:

```text
update where id + expected state + scope
if affected count is zero -> conflict
```

Exact implementation depends on Prisma schema and repository style.

### Optimistic Version Check

Use when schema has a version or updatedAt-based concurrency strategy approved.

Expected behavior:

```text
client or service provides expected version
write succeeds only when version matches
mismatch returns conflict
```

### Unique Constraint Conflict

Use when duplicate create must be prevented by schema truth.

Expected behavior:

```text
constraint violation maps to conflict
service documents whether duplicate intent should replay, conflict, or return existing state
```

### Transaction Serialization

Use when multiple related writes must be protected as a unit.

Expected behavior:

```text
transaction boundary wraps all dependent writes
conflict remains explicit when state is no longer valid
```

## FieldOps Candidate Application

### Equipment create

Conflict review:

```text
REQUIRED
```

Reason:

```text
code or qrCode duplicate behavior must be explicit and schema-backed.
category auto-create path may race with another request creating the same category.
```

Expected follow-up:

```text
verify equipment and equipmentCategory unique constraints in Prisma schema
map duplicate create to conflict or safe replay based on product contract
```

### Equipment update

Conflict review:

```text
REVIEW
```

Reason:

```text
basic field overwrite may be acceptable, but category reassignment and status/condition update can depend on current state.
```

### Equipment status update

Conflict review:

```text
REQUIRED
```

Reason:

```text
status change is a lifecycle-style transition guarded by policy.
competing updates may produce stale transition decisions.
```

### Equipment soft delete

Conflict review:

```text
REVIEW NOW / REQUIRED WHEN RELATIONS EXIST
```

Reason:

```text
soft delete is simple today, but becomes consistency-sensitive when equipment is linked to active sessions, movements, checkout, required tools, or return verification.
```

## Workflow Module Defaults

The following planned workflow modules should default to conflict review before implementation completion:

```text
field-session
issue
attachment
site-exit-verification
return-verification
billing
expense
notification
```

High-risk operations include:

```text
field-session close / reopen
issue assign / escalate / resolve
site-exit verification submit
return verification finalize
billing finalize
expense approve / reject
payout create / mark paid
notification intent de-duplication
```

## Verification Requirements

Conflict-sensitive operations must provide one or more tests or documented deferrals:

```text
stale state returns conflict
competing update does not silently overwrite lifecycle state
duplicate create maps to expected conflict or replay behavior
organization scope prevents cross-org conflict leakage
transaction rollback preserves prior state when conflict occurs mid-flow
```

## Deferral Format

If conflict handling is deferred, record:

```text
operation
risk
why deferred
blocking schema/product/test dependency
owner
next milestone
```

## Implementation Gate

Do not implement conflict runtime changes until:

```text
operation-level transition rules are known
schema support is verified
error contract is confirmed
test strategy is available
transaction boundary relationship is clear
```
