# BE-08.03 Unit-of-Work Foundation

Status: Draft
Phase: BE-08 Transaction and Consistency
Milestone: BE-08.03
Owner: BE-08 Transaction and Consistency Owner

## Purpose

This document defines the Unit-of-Work foundation for FieldOps Backend.

The goal is to provide a domain-neutral convention for executing transaction-bound service work while preserving Backend ownership boundaries.

## Core Principle

Unit-of-Work is an execution boundary, not a business layer.

```text
Service owns business decision
Unit-of-Work owns transaction execution mechanics
Repository owns persistence operation
Prisma owns database client behavior
```

A Unit-of-Work helper must make transaction use consistent without hiding domain decisions.

## Public Contract

Preferred helper contract:

```text
runInTransaction(async ({ client }) => {
  // service-owned business flow
})
```

Minimum behavior:

```text
accepts an async callback
opens exactly one Prisma transaction
passes transaction client to callback
returns callback result
rolls back when callback throws
keeps original error intent where possible
```

## Service Usage Rule

Services may use Unit-of-Work when BE-08.02 required transaction rules apply.

Example shape:

```text
async function createSomething(input) {
  validate(input)
  assertScope(input.organizationId)

  return runInTransaction(async ({ client }) => {
    const first = await repository.createFirst({ ...input, client })
    const second = await repository.createSecond({ firstId: first.id, client })
    return mapResult(first, second)
  })
}
```

Business decisions remain in the service.

## Repository Client Rule

Repository methods that may be used in a transaction should accept a transaction client.

Preferred object-argument form:

```text
repository.createThing({ organizationId, data, client })
```

Existing positional form may use:

```text
repository.updateThing(organizationId, id, data, { client })
```

Repository methods must use the provided client when supplied and default to the root Prisma client otherwise.

## Forbidden Unit-of-Work Behavior

The Unit-of-Work helper must not:

```text
perform validation
choose organization scope
make policy decisions
map domain responses
create domain-specific records by itself
know module lifecycle states
swallow errors silently
convert all errors to generic errors
start nested transactions without explicit support
```

## Nested Context Rule

Nested transaction context is review-only until a standard exists.

Default rule:

```text
Do not call runInTransaction from inside another runInTransaction block.
```

If a service must compose multiple operations, prefer extracting inner domain operations that accept `{ client }` instead of opening their own transaction.

## Error Propagation Rule

When callback throws:

```text
Prisma transaction rolls back
original domain error should be preserved
repository errors should remain mapped through repository error mapping
unexpected errors should flow to normal backend error handling
```

The Unit-of-Work helper should not own business error translation.

## Organization Scope Rule

Organization scope must be validated before entering the Unit-of-Work or at the first service-owned guard inside the callback.

Inside the callback:

```text
all scoped repository calls must pass organizationId
created records must include organizationId where required
read-before-write checks must preserve organizationId where applicable
```

## Side-Effect Intent Rule

If the operation needs audit/outbox/intent records, they should be written inside the same Unit-of-Work when they represent part of the same consistency outcome.

Dispatch or external execution should happen outside the transaction from persisted intent.

## Equipment Candidate Application

The current equipment create flow has a candidate multi-write path:

```text
find category by code
create category if missing
create equipment
```

Possible BE-08.03-compatible refactor shape:

```text
return runInTransaction(async ({ client }) => {
  const categoryId = await findOrCreateCategoryByCode({ organizationId, categoryCode, client })
  return equipmentRepository.createRuntimeEquipment({ data: { ...data, categoryId }, client })
})
```

This is a proposed shape only. Implementation requires schema constraint review and test coverage.

## Test Expectations

A Unit-of-Work implementation requires tests for:

```text
commits successful callback result
rolls back when callback throws
passes transaction client to callback
preserves thrown domain error
prevents or documents nested transaction behavior
```

Service-level adoption requires tests for:

```text
multi-write success
rollback on second write failure
organization scope preservation
conflict behavior when applicable
```
