## Business Workflow Before

The Auth module was already partially normalized:

- controller delegated to service;
- service owned login, registration, and current-user session workflows;
- policy owned user active checks and active membership selection;
- repository owned data access and transaction wrapper.

However, login user-selection rules for zero users and multi-organization email conflict still lived directly in the service.

## Business Workflow After

Auth policy now exposes:

```text
assertSingleLoginUser
assertUserExists
assertUserActive
findActiveMemberForUser
```

Auth service now delegates login user-selection to `assertSingleLoginUser` and remains responsible for orchestration:

- fetch login candidates from repository;
- assert selected login user;
- verify password;
- find active membership;
- sign access token;
- build auth session response shape.

## API Contract

No intentional API contract change.

Existing service method names, controller behavior, error classes, and response mapping flow were preserved.

## Repository Boundary

No repository source was changed.

Repository remains data-access only and exposes transaction wrapper for registration.

## Transaction Boundary

`registerOrganizationOwner` still uses repository transaction wrapper to create organization, user, and organization member in one transaction.

BE-08 should later review registration transaction consistency and uniqueness/conflict behavior under concurrent registration attempts.

## Validation Boundary

Request validation was not introduced or changed.

Broader login/register validation belongs to BE-06.

## Policy Boundary

Policy now owns:

- single login user selection;
- invalid credential result for missing login candidate;
- multi-organization login conflict;
- user active check;
- active membership selection.

## Audit / Observability Boundary

No audit behavior was introduced.

BE-09 may later consider audit intent for login success/failure, owner registration, and current-user access.
