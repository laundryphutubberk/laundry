## Standard Responsibility Split

```text
Route
  owns URL path, HTTP method, middleware chain, auth and organization guard composition

Controller
  owns request/response orchestration only

Service
  owns business execution decisions, organization-scoped access, mutation rules, response mapping intent, domain errors, and transaction decisions

Repository
  owns data-access only
```

## Auth Module Baseline Observation

### Controller

`fieldops-be/src/modules/auth/auth.controller.js` currently:

- imports HTTP response helper through `success`;
- reads `req.body` and `req.user.id`;
- calls `authService.login`, `authService.registerOrganizationOwner`, and `authService.getCurrentUser`;
- maps output through response mapper functions;
- returns HTTP status codes and response envelopes.

This aligns with the controller responsibility boundary.

### Service

`fieldops-be/src/modules/auth/auth.service.js` currently owns meaningful business workflow:

- login credential flow;
- duplicate email / multi-organization conflict behavior;
- active user and active membership checks;
- access-token payload construction;
- organization-owner registration workflow;
- current-user session assembly;
- auth session shaping before response mapping.

This mostly aligns with BE-05 service orchestration ownership.

### Repository

`fieldops-be/src/modules/auth/auth.repository.js` currently owns data-access methods only:

- user lookup by email;
- user lookup by id;
- transaction wrapper;
- organization creation;
- user creation;
- organization-member creation;
- database ping.

No explicit business decision was observed inside the repository during this inspection.

## BE-05 Gaps Observed

### Transaction Boundary

`registerOrganizationOwner` uses a repository transaction wrapper around organization, user, and organization-member creation.

This is a transaction boundary that should be recorded for BE-08 Transaction and Consistency review. BE-05 may keep the orchestration shape but should avoid changing transaction implementation unless explicitly required.

### Policy Boundary

Auth status and membership checks are currently implemented directly inside service helpers.

This is acceptable for current BE-05 source reality, but policy extraction or reusable policy formalization should be recorded for BE-07 if it becomes shared across modules.

### Validation Boundary

Login and registration request validation are not owned by BE-05.

BE-05 should not introduce request validation as a primary concern. Required validation gaps should be recorded for BE-06.

### Audit Boundary

Auth registration contains runtime console logging. BE-05 should not convert this into observability infrastructure unless routed. Audit intent and observability gaps should be recorded for BE-09.
