## Business Workflow Before

`fieldSession.controller.js` already stayed mostly HTTP-focused by building user context, calling service methods, applying response mappers, and returning response envelopes.

`fieldSession.service.js` owned lifecycle orchestration, including:

- active session resolution
- session listing and detail lookup
- session creation
- session update
- archive
- status transition
- close
- scan item workflow
- equipment movement creation through scan workflow

However, the service directly created organization context, session id, not-found, and closed-session errors in multiple workflow paths.

## Business Workflow After

`fieldSession.policy.js` now exposes:

```text
assertFieldSessionOrganizationContext
assertFieldSessionId
assertFieldSessionExists
assertFieldSessionEditable
```

`fieldSession.service.js` now delegates reusable business guards to policy and remains responsible for use-case orchestration.

The lifecycle guard/policy files remain responsible for transition rules:

```text
fieldSessionLifecycle.policy.js
fieldSessionLifecycle.guard.js
```

## Policy Boundary

Field Session policy now owns:

- organization context requirement
- session id requirement
- session existence assertion
- editability guard for closed sessions

Lifecycle guard/policy continues to own:

- valid session status transitions
- valid session item status transitions
- scan mode/session status compatibility
- close readiness based on item statuses

## Transaction Boundary

Existing transaction boundaries were preserved:

- create field session with participants and planned items
- close field session
- scan session item and create equipment movement

BE-08 should later review whether scan workflow should also update equipment runtime status atomically with session item and movement records.

## Audit / Observability Boundary
