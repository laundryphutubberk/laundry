## Intended Future Service Ownership

When Job Finance modules are introduced, service-layer use cases should own workflows such as:

```text
createJobBillingDraft
calculateJobBillableItems
approveJobBilling
recordJobExpense
approveJobExpense
calculateTechnicianPayout
approveTechnicianPayout
settleJobFinance
getJobFinanceSummary
```

These workflows should coordinate domain data but keep business decisions out of controllers and repositories.

## Future Module Boundary

A future finance implementation should separate responsibilities as follows:

```text
Job service
  owns job lifecycle and job completion readiness

Billing service
  owns billable item calculation, invoice draft intent, billing approval workflow

Expense service
  owns job expense recording, approval workflow, and reimbursement readiness

Payout service
  owns technician payout calculation and settlement readiness

Repository layer
  owns data access only

Controller layer
  owns HTTP request/response orchestration only
```

## Cross-Domain Dependencies

Future Job Finance workflow may depend on:

```text
field-session lifecycle

service item and consumable usage
equipment movement evidence
```

BE-05 should coordinate these through explicit service use cases, not through direct controller-to-repository orchestration.
