# ADR-0002 — Backend Transaction and Inventory Projection Standard

Status: Accepted

## Context

BE-08 Transaction and Consistency requires a backend-wide standard for transaction boundaries, idempotency, concurrency, repository transaction support, side effects, and inventory projection before implementation starts.

The backend domain has workflows that can update multiple models in one business operation, including Laundry Work, Laundry Bag, Laundry Count Line, Issue Report, Linen Movement, Linen Inventory Summary, Work Status Log, and Wash Load Plan.

Partial commits in these workflows can create inconsistent states such as:

- Laundry Bag created without Laundry Work status/bag count update.
- Laundry Count Line created without Linen Movement.
- Linen Movement created without Inventory Summary projection.
- Status changed without Work Status Log.
- Issue Report created without related movement or issue count consistency.

The Business Blueprint and domain model establish these principles:

- Laundry Work is the operational center.
- Laundry Bag is the intake unit.
- Laundry Count Line is the real count captured at laundry.
- Linen Movement is the source of inventory movement truth.
- Linen Inventory Summary is derived from movement history.
- Resort Workspace must remain isolated by resortId.
- Issues must be explicit.

## Decision

### 1. Transaction Standard

Backend uses service-owned interactive transactions for all multi-write business workflows.

```text
Service
  -> prisma.$transaction(async tx => ...)
  -> repository methods receive shared transaction client
  -> all writes in the workflow commit or rollback together
```

A transaction is required when an operation:

- writes more than one model;
- writes a main entity and a history/log record;
- writes a ledger/movement and a derived summary;
- updates status and writes a status log;
- changes an aggregate and one of its child entities in the same business operation;
- performs a retry-sensitive workflow where partial commit would create drift.

A transaction is not required for:

- read-only queries;
- single write operations with no dependent side effect;
- dashboard/report reads;
- lookup/master-data reads.

### 2. Repository Transaction Standard

Repositories that participate in write workflows must accept an optional Prisma transaction client through a `client` parameter.

```text
function repositoryMethod({ ..., client } = {}) {
  const db = client || prisma;
}
```

Services own workflow orchestration. Repositories own Prisma query shape only.

Controllers must not import Prisma.
Services should not import Prisma directly except through an approved transaction helper or repository transaction wrapper.

### 3. Idempotency Standard

Backend uses layered idempotency.

For MVP:

- Use natural business keys where already defined.
- Use duplicate detection before create.
- Use database unique constraints where available.
- Use expected state checks for status transitions.

Natural keys include:

- `LaundryWork.workNo` for Create Work.
- `LaundryBag`: `workId + bagNo` for Create Bag.

For retry-sensitive future workflows, add explicit idempotency keys before public production exposure:

- Count Linen.
- Issue Report.
- Return Workflow.
- Inventory-affecting Movement creation.

### 4. Concurrency Strategy

Backend uses optimistic concurrency as the baseline.

Status transitions should use expected-current-state guards where possible.

```text
update where:
  id = targetId
  currentStatus = expectedStatus
```

If the update affects no row, the operation must be treated as a conflict or stale-state request.

Pessimistic locking or serializable transactions are not the default. They may be introduced only after evidence shows that optimistic concurrency is insufficient for a specific workflow.

### 5. Inventory Projection Strategy

For MVP, Inventory Summary uses synchronous projection inside the same transaction.

```text
Count / Issue / Return
  -> LinenMovement
  -> LinenInventorySummary
```

`LinenMovement` remains the source of truth.
`LinenInventorySummary` is a derived read model.

Inventory Summary must not be edited directly by public CRUD flows.

A future ADR may replace synchronous projection with an asynchronous projection job or outbox pattern if scale, latency, or operational needs require it.

### 6. Side Effect Strategy

Side effects are classified as follows:

| Side Effect | Transaction Decision |
|---|---|
| WorkStatusLog | In transaction |
| LinenMovement | In transaction |
| LinenInventorySummary | In transaction for MVP |
| Business audit required for correctness | In transaction |
| Notification | After commit |
| Email / SMS / webhook | After commit |
| External event publication | After commit or future outbox |
| Cache invalidation | After commit |
| Analytics/metrics | After commit |

## Consequences

### Positive

- Reduces partial commit risk.
- Makes transaction boundaries explicit and reviewable.
- Keeps business workflow orchestration in the service layer.
- Allows repositories to be reused inside and outside transactions.
- Keeps Linen Movement as the inventory source of truth.
- Keeps Inventory Summary consistent during MVP without adding worker infrastructure.

### Trade-offs

- Multi-write services must be carefully scoped to avoid long transactions.
- Retry-sensitive commands still need explicit idempotency keys before production hardening.
- Optimistic concurrency may require additional implementation work in status update flows.
- Future async projection/outbox strategy may require another ADR.

## Implementation Notes

BE-08 implementation should proceed in small gates:

1. Confirm and normalize repository transaction-client support.
2. Add optimistic status transition guards where needed.
3. Implement Count Linen transaction boundary.
4. Implement Issue Report transaction boundary.
5. Implement Return Workflow transaction boundary.
6. Implement synchronous Movement -> Inventory Summary projection.
7. Add idempotency support for retry-sensitive workflows.
8. Add transaction and consistency verification tests.

## Stop Conditions

Stop and request architecture review if implementation requires:

- Business Blueprint change.
- schema.prisma change.
- API Contract change.
- Workspace Boundary change.
- Permission model change.
- Technology baseline change.
- Aggregate Boundary change.
- Moving Inventory Summary projection to async job/outbox.
